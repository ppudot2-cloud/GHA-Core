Power Platform CI/CD — End-to-End Pipeline Flow
GHA-Core + GHA-Dynamics · Two-Pipeline Architecture · Azure OIDC + AKV
Pipeline 1: Build & Deploy Pipeline 2: UAT + Prod Azure OIDC · No Secrets in GitHub Mock Deploy Supported

🏗️ Two-Repo Architecture

  • GHA-Core — Reusable library: composite actions, reusable workflows, PowerShell scripts
  • GHA-Dynamics — Project caller: trigger workflows, solutions.json, deployment-settings, solution source
  • GHA-Core is checked out at .ci/ on every runner by reveille
  • All business logic lives in GHA-Core — GHA-Dynamics is a thin wrapper

🔐 Authentication — Zero Secrets in GitHub

  • Only one GitHub secret needed: GHA_CORE_PAT (to checkout GHA-Core)
  • Azure login via OIDC / Workload Identity Federation — no client secret stored
  • PP credentials (pp-app-id, pp-client-secret, pp-tenant-id) fetched live from Azure Key Vault
  • GitHub Variables (non-sensitive): AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_SUBSCRIPTION_ID, AZURE_KEY_VAULT_NAME

📦 Two-Pipeline Architecture

  • Pipeline 1 (build-and-deploy.yml) — triggered by workflow_dispatch or push to feature/**. Export stage writes pipeline-context.json to the feature branch → Build → Deploy Dev/Intg/UAT/FRS/Perf in parallel → create PR to main. Use skip_export=true (or push a manually-exported branch) to bypass sandbox export.
  • Pipeline 2 (deploy-prod.yml) — triggered automatically when PR merges to main. Reads pipeline-context.json → Deploy UAT (final gate) → Deploy Prod
  • Cross-pipeline artifact handoff via pipeline-context.json on main branch

🔄 Platform Constraints

  • Sequential imports are mandatory — Dataverse cannot process parallel solution imports to the same environment
  • deploy-all-solutions action loops solutions in deployOrder sequence from solutions.json
  • Different environments run in parallel (Pipeline 1), but within each environment solutions are sequential
  • Workflows must live in .github/workflows/ root — GitHub does not support subdirectories

Pipeline 2 deploy-prod.yml — UAT Re-validation + Production

Triggered automatically when a feature/* branch PR is merged to main (i.e. the pull_request closed+merged event). The guard job verifies pull_request.merged == true and the head branch starts with feature/. Downloads build artifacts from Pipeline 1 using the run_id stored in pipeline-context.json.

#Job / StepTypeWhat it does
1guardinlineConfirms trigger is a merged PR from a feature/* branch or manual dispatch. Only proceeds if pull_request.merged == true and head branch starts with feature/. Logs PR number, branch, commit, and actor.
2read-contextinlineChecks out main, parses pipeline-context.json → outputs run_id, run_number, matrix, solution_list for downstream jobs
3deploy-uatenv gateUses environment: UAT (approval gate). reveille → pac-install → download artifacts from Pipeline 1 run via run-id → deploy-all-solutions (UAT, Managed)
4deploy-prodenv gateOnly if UAT succeeded. Uses environment: Prod (approval gate). Same steps → deploy-all-solutions (Prod, Managed, import_config_data=true, tag_prod_deployed=true)
5pipeline-summaryinlineAlways runs. Writes summary table, sends failure email notification.
Why UAT is deployed twice: Pipeline 1 deploys UAT as a non-prod validation step (along with Dev/Intg/FRS/Perf in parallel). Pipeline 2 re-deploys UAT as the final gate before Prod — ensuring the artifacts that will hit Prod are proven on UAT in the same pipeline run that precedes Prod approval.

Automatic Rollback Built into deploy-all-solutions

There is no standalone rollback workflow. Recovery is fully automatic inside the deploy pipeline.
ConditionBehaviour
enable_backup=true + upgrade + import failsPipeline immediately re-imports backup/{name}_{env}_backup.zip to restore the previous version. Logged clearly in Step Summary. No manual action needed.
enable_backup=false + import failsJob fails. Environment left in partial state. Re-run the pipeline with a fix.
First install + import failsNo rollback — there is no previous version to restore to. Solution is simply absent from the environment.
Auto-rollback also failsBoth failures are logged. Environment flagged as needing manual intervention.
Recommendation: Set enable_backup=false for Dev and Intg (re-run is sufficient), and enable_backup=true for UAT, FRS, Perf, and Prod.

All composite actions live in GHA-Core/.github/actions/dynamics/. They are referenced as ppudot2-cloud/GHA-Core/.github/actions/dynamics/{name}@main.

reveille

.github/actions/dynamics/reveille/action.yml

Shared first step for every job. Checks out caller repo (full history), checks out GHA-Core to .ci/, performs Azure OIDC login, fetches AKV secrets (PP credentials, optionally JFrog token), merges global-vars.yml + project-vars.yml into $GITHUB_ENV. Skips Azure/AKV steps in mock mode.

pac-install

.github/actions/dynamics/pac-install/action.yml

Installs Microsoft Power Platform CLI via microsoft/powerplatform-actions/actions-install@v1 and adds to PATH. No inputs required.

pack-solution

.github/actions/dynamics/pack-solution/action.yml

Reads version from Solution.xml, stamps it to Major.Minor.RunNumber.Attempt, strips the <Managed> tag, packs both Unmanaged and Managed ZIPs (via PAC CLI or mock simulation). Outputs: version, artifact_name, unmanaged_zip, managed_zip, out_dir.

solution-checker

.github/actions/dynamics/solution-checker/action.yml

Runs PAC Solution Checker against the unmanaged ZIP for the specified geography. Generates SARIF output. Uploads checker artifact. Always mandatory in real mode; simulated in mock mode via Invoke-SolutionCheckerSim.ps1.

export-config-data

.github/actions/dynamics/export-config-data/action.yml

Exports configuration migration data from the source PP environment using the provided schema XML file. Writes output to config-data/{name}-data.zip. Skips if no schema file provided. Simulated in mock mode via Export-ConfigDataSim.ps1.

export-solution

.github/actions/dynamics/export-solution/action.yml

Exports an unmanaged solution from a PP environment using PAC CLI. Used by the export stage in _stage-export.yml.

import-solution

.github/actions/dynamics/import-solution/action.yml

Wraps PAC solution import for all variants: unmanaged, managed, and stage-and-upgrade (holding solution pattern). Handles deployment settings, async wait, plugin activation.

deploy-all-solutions

.github/actions/dynamics/deploy-all-solutions/action.yml

The main deploy orchestrator. Loops through all solutions in solutions_json in deployOrder sequence. For each solution: token substitution → base check → blocking check → version compare → find solution → backup → import → config data → publish → activate flows → JFrog tag → deploy summary. Uploads backup artifact after loop.

pre-deploy-checks

.github/actions/dynamics/pre-deploy-checks/action.yml

Runs pre-import validation: Invoke-BlockingCheck.ps1 (abort if async operations in progress) and Compare-SolutionVersion.ps1 (verify version progression is correct).

post-deploy

.github/actions/dynamics/post-deploy/action.yml

Post-import actions: tags the JFrog artifact as prodDeployed=true (Prod only), writes per-solution deploy summary via Write-DeploySummary.ps1.

jfrog-upload

.github/actions/dynamics/jfrog-upload/action.yml

Uploads managed ZIP, unmanaged ZIP, and SARIF to JFrog Artifactory for long-term archival. Runs once per build. Not used for artifact download during deploys (GitHub artifact storage is used for that).

Key Configuration Files

solutions.json

GHA-Dynamics/solutions.json

Single source of truth for the solution registry. Defines name, source folder, deploy order, deployment settings paths per environment, and config migration schema. Read by Resolve-SolutionMatrix.ps1 to build the GitHub Actions matrix.

pipeline-context.json

GHA-Dynamics/pipeline-context.json

Written by the export stage (_stage-export.yml) commit job during Pipeline 1 — before the build and deploy stages. Contains runId, runNumber, runAttempt, matrix, solutionList, featureBranch, triggeredBy, exportMode. Committed to the feature branch; lands on main when the PR is merged. Pipeline 2 uses runId to download artifacts from Pipeline 1.

deployment-settings/{env}/{SolutionName}.json

GHA-Dynamics/deployment-settings/

Per-solution, per-environment variable overrides injected at import time. Contains EnvironmentVariables and ConnectionReferences. Token placeholders #{TOKEN}# are replaced at runtime from GitHub Variables/Secrets by deploy-all-solutions.

config/{SolutionName}/data-schema.xml

GHA-Dynamics/config/

Configuration Migration Tool schema XML defining which entities and records to export/import. Referenced in solutions.json via dataSchemaFile. Used by export-config-data action and imported by deploy-all-solutions when import_config_data=true.

global-vars.yml

GHA-Core/.github/config/global-vars.yml

Org-wide default variable values. Merged first by Merge-Variables.ps1. Project-level values in project-vars.yml override these. Ensures consistent defaults across all GHA-Dynamics repos.

project-vars.yml

GHA-Dynamics/.github/config/project-vars.yml

Project-specific variable overrides. Values here take precedence over global-vars.yml. Used for project-specific feature flags, timeouts, or config values that vary per project.
