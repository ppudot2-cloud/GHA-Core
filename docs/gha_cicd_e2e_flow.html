🌅 Reveille — Bootstrap Action
.github/actions/dynamics/reveille/action.yml · Called by every job as step 1
▼
R1
Checkout calling repository ▶
actions/checkout@v4 with fetch-depth: 0. Checks out GHA-Dynamics so scripts can access solutions.json, deployment-settings/, etc.
R2
Checkout GHA-Core CI scripts ▶
actions/checkout@v4 → ppudot2-cloud/GHA-Core@main, auth via env.GHA_CORE_PAT, path .ci/. Makes available: 14 PS scripts, global-vars.yml, ServiceNow PS module.
R3
Azure Login (OIDC) skipped in mock ▶
azure/login@v2 — Workload Identity Federation. GitHub issues a short-lived JWT → Azure AD validates against Federated Identity Credential → issues temporary access token valid only for the runner job duration. No client secret stored in GitHub.
R4
Fetch secrets from Azure Key Vault skipped in mock ▶
Always: pp-app-id → PP_APP_ID, pp-client-secret → PP_CLIENT_SECRET, pp-tenant-id → PP_TENANT_ID.
If jfrog_enabled=true: jfrog-api-key → JFROG_TOKEN.
If mulesoft_enabled=true: mulesoft-client-id/secret → MULESOFT_CLIENT_ID/SECRET.
If servicenow_enabled=true: snow-base-uri → SERVICENOWMURI, snow-oauth-client-id/secret → SNOW_OAUTH_*.
Each value is masked (::add-mask::) before writing to $GITHUB_ENV — never appears in logs.
R5
Merge global + project variables ▶
Merge-Variables.ps1 reads global-vars.yml (GHA-Core defaults) + project-vars.yml (GHA-Dynamics overrides).
Checks each project key against protected_keys list — fails immediately if project tries to override a protected key.
Azure identity keys (AZURE_*) excluded from merge (OIDC already set them).
Writes all merged KEY=value pairs to $GITHUB_ENV.
R6
Register JFrog PS module repo if jfrog_enabled ▶
Unregisters PSGallery (no internet in restricted environments). Registers JFrog Artifactory NuGet v2 feed as trusted PS repository named "JFrog" using JFROG_TOKEN + JFROG_URL/JFROG_REPO. All subsequent Install-Module calls resolve from JFrog.
🔄 Deploy All Solutions
.github/actions/dynamics/deploy-all-solutions/action.yml · Called by every deploy job as step 4
▼
Deploys every solution in solutions_json to ONE environment in deployOrder sequence. Pre-conditions: reveille, pac-install, download-artifact must have already run.
Informational only — never fails the pipeline. Pre-flight checks run before ServiceNow so the user can see environment health in the Actions log before a CR is opened. Any issues (missing base solution, blocking op, artifact not found) are logged as ⚠️ warnings and the deployment proceeds. If the environment truly cannot accept the import, the actual pac solution import in the deploy loop will surface the real Dataverse error with full context.
STEP A — PAC Auth
A
PAC auth — target environment ▶
Real: pac auth create --name target-<env> --url <env_url> --applicationId PP_APP_ID --clientSecret PP_CLIENT_SECRET --tenant PP_TENANT_ID → pac auth select --name target-<env>. Needed before pre-flight because base-solution check uses pac solution list. Mock: logs skip.
STEP B — Pre-flight Loop (ALL solutions · informational · never fails pipeline)
Pre-flight checks — all solutions in deployOrder · warns on issues · always continues · writes preflight-results.json
1
Verify artifact present ▶
Real: checks out/<name>/<name>-<version>_managed.zip exists. Logs ⚠️ warning if missing (download-artifact may have failed or artifact name mismatch). Deployment continues regardless. Mock: creates placeholder stub ZIP.
2
Resolve deployment settings tokens ▶
Reads deployment-settings/<env>/<name>.json. Scans for #{TOKEN_NAME}# patterns, reads $env:TOKEN_NAME for each match, writes resolved JSON to out/<name>/deployment-settings-resolved.json. Mock: skips resolution.
3
Verify base solutions if base_solutions set ▶
pac solution list --environment <url> for each base solution name. Logs ⚠️ warning if any base solution is missing — deployment continues, but import will likely fail with a Dataverse dependency error.
4
Blocking check if enable_blocking_check ▶
Invoke-BlockingCheck.ps1 queries Dataverse for in-progress async operations (imports, upgrades, publishes). Fails if blocking operations found — no SNOW CR opened, saves approvers from approving a deployment that can't proceed yet.
5
Version compare if enable_version_compare ▶
Compare-SolutionVersion.ps1: reads artifact version from filename; queries Dataverse API on previous_environment_url and target_environment_url. Sets skip_import=true if artifact ≤ target version (no SNOW needed). Returns 0.0.0.0 (non-fatal log) if previous env never deployed.
✓
Writes preflight-results.json with per-solution skip_import flags + resolved paths
STEP C — ServiceNow Pre-Deploy (always reached after pre-flight · informational checks never block)
C
ServiceNow Pre-Deploy if SNOW enabled · BLOCKS ▶
Calls servicenow-change@main phase: pre-deploy. Opens CR, sets change window, requests approval, polls until approved. Pipeline is blocked here until SNOW approver approves the CR. Only runs if enable_servicenow=true. Always reached — pre-flight warnings do not gate this step. See ServiceNow CR Lifecycle reference for all 8 sub-steps.
STEP D — Deploy Loop (ALL solutions · SNOW approved · try/catch with inline rollback)
Deploy loop — reads preflight-results.json for skip_import + resolved settings
6
Find solution in target environment ▶
pac solution list --environment <url> --json. Parses JSON to find solution by SolutionUniqueName. If found: sets $solutionExists=true. Determines import mode: UPGRADE (holding pattern) vs INSTALL (standard). Mock: treats as first install.
7
Backup upgrades only · if enable_backup ▶
Only on upgrades (solutionExists=true, not skip_import). Skipped on first installs. Real: pac solution export --name <name> --path backup/<name>_<env>_backup.zip --managed. Used by catch block for inline rollback.
8
Import solution ▶
UPGRADE: pac solution import --import-as-holding + pac solution upgrade --async.
INSTALL: pac solution import --force-overwrite (+ optional --settings-file from pre-flight resolved path).
skip_import=true (set by version compare in pre-flight): logs "version already current", skips.
Mock: logs simulated import with mode and settings file info.
9
Config data import PROD ONLY ▶
Only if import_config_data=true AND not skip_import AND config-data/<name>-data.zip exists. pac data import --path config-data/<name>-data.zip --environment <url>.
10
Publish customizations ▶
Skipped if skip_import or UPGRADE mode (upgrade publishes atomically). Real: pac solution publish --environment <url>. Mock: logs simulation.
11
Extract + activate flows & workflows ▶
Part A: Expand-Archive managed ZIP to out/<name>/_extracted/.
Part B — Cloud flows (.json): scans _extracted/Workflows/*.json; extracts flow GUID from filename; calls pac flow enable --flow-id <guid> --environment <url> for each.
Part C — Classic workflows (.xaml): scans _extracted/Workflows/*.xaml; reads workflow Id attribute; acquires bearer token via client_credentials OAuth; calls Dataverse Web API PATCH /api/data/v9.2/workflows(<id>) with {"statecode":1,"statuscode":2}.
POST-DEPLOY
C
Catch — Inline Rollback on exception ▶
Writes SNOW_DEPLOY_STATUS=failure to $GITHUB_ENV (read by SNOW post-deploy step). If enable_backup=true AND backup ZIP exists (upgrades only): pac solution import --path backup/<name>_<env>_backup.zip --force-overwrite. Adds to $failed list. After loop: if $failed.Count > 0, exits 1.
S
Write SNOW_DEPLOY_STATUS=success ▶
Written to $GITHUB_ENV after loop completes without error. Post-deploy SNOW step reads this to determine whether to close the CR as successful or unsuccessful.
SN
ServiceNow Post-Deploy if SNOW · always()
→ See ServiceNow CR reference (Steps 12–14)
BU
Upload backup artifact if enable_backup ▶
Artifact name: backup-<env>-v<run_number>. Path: backup/. Retention: 30 days. Uploaded regardless of deploy outcome for audit purposes.
❄️ ServiceNow CR Lifecycle
.github/actions/dynamics/servicenow-change/action.yml · Called by deploy-all-solutions when SNOW enabled
▼
Called twice: pre-deploy (8 steps, blocks until approved) + post-deploy (3 steps, if: always()). Credentials: SERVICENOWMURI, SNOW_OAUTH_CLIENT_ID/SECRET set by Reveille when servicenow_enabled=true.
PRE-DEPLOY (8 steps — pipeline blocked until approved)
SN1
Load ServiceNow PS module ▶
Import-Module .ci/.github/servicenow/ServiceNow.psd1 — makes all SNOW cmdlets available for subsequent steps.
SN2
Set runtime environment variables ▶
Sets BUILD_UNIQUE_IDENTIFIER = <repo>-<run_number>-<environment>. Sets SERVICENOWSHORTDESCRIPTION = Deploy <solutions> to <env> — run #N. Calculates change window from SERVICENOW_DESIRED_DAY variable. Writes all to $GITHUB_ENV.
SN3
New-ServiceNowChangeRequest ▶
Creates CR in ServiceNow. Populates fields from env vars: SERVICENOWCHANGETYPE, SERVICENOWASSIGNMENTGROUP, SERVICENOWJUSTIFICATION, SERVICENOWIMPLEMENTATIONPLAN, SERVICENOWBACKOUTPLAN, SERVICENOWRISKLEVEL, SERVICENOWIMPACTLEVEL, etc.
Writes SNOW_CHANGE_REQUEST_NUMBER (e.g. CHG0123456) and SNOW_CHANGE_REQUEST_ID (sys_id GUID) to $GITHUB_ENV.
SN4
Add-ServiceNowAuditTrailArtifact ▶
Attaches Solution Checker SARIF file to the CR as an audit trail artifact. Only runs if sarif_path input is non-empty. Satisfies audit evidence requirements.
SN5
Set-ServiceNowChangeWindow ▶
Sets planned start/end time on the CR using the calculated change window. Satisfies ITSM governance requirements for scheduled change windows.
SN6
Get-ServiceNowConflict ▶
Queries ServiceNow for scheduling conflicts with the planned change window. Logs conflicts as warnings — does not block (conflict handling is a business process decision).
SN7
Request-ServiceNowApproval ▶
Moves CR from New → Awaiting Approval. Triggers notification to approvers in ServiceNow.
SN8
Get-ServiceNowApprovalStatus BLOCKS pipeline ▶
Polls ServiceNow on a configurable interval until CR is approved, rejected, or timed out. Interval and timeout controlled by SERVICENOW_APPROVAL_TIMEOUT_MINUTES.
Approved: logs approval, writes approver info to env, pipeline continues.
Rejected: writes SNOW_DEPLOY_STATUS=rejected to $GITHUB_ENV, fails the step, blocks the deploy.
Timeout: fails the step after the configured timeout.
POST-DEPLOY (3 steps — if: always())
SN12
Find GitHub Environment approvers ▶
GitHub Actions REST API: GET /repos/{owner}/{repo}/actions/runs/{run_id}/approvals. Reads list of users who approved the GitHub Environment gate. Falls back to GITHUB_ACTOR if no approvals found.
SN13
Read SNOW_DEPLOY_STATUS ▶
Reads SNOW_DEPLOY_STATUS from $GITHUB_ENV. This variable was written before any exit 1 in the deploy loop — always reflects the true deploy outcome. Value: success, failure, or rejected.
SN14
Close-ServiceNowChangeRequest ▶
Close-ServiceNowChangeRequest with: close_code: successful (if status=success) or unsuccessful (if failure/rejected). Passes CR number, sys_id, and approver info. Moves CR to Closed state in ServiceNow with appropriate close notes.
